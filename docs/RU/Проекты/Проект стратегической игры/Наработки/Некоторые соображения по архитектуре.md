Аксиома для любого веполя:
$$\forall v \vee v_i \exists p: v \rightarrow p \rightarrow v_i$$

Соответственно, каждая вершина и каждый протокол должны наследовать интерфейсы (а не реализацию), чтобы иметь соответствующие сигнатуры для обмена сообщениями. С другой стороны, описать одинаково хорошо одной реализацией нацию и танк не выйдет. Значит, протоколы надо реализовать, как адаптеры с двумя маппингами (для прямой и обратной связи), а вершины - как состояния, чьё поведение будет зависеть от битовой маски (которая, в свою очередь, будет меняться самими же вершинами в зависимости от сигналов протоколов).

## Архитектура игровой сущности и протоколов взаимодействия

### 1. Игровая сущность (Entity)

Базовый принцип: каждая игровая сущность является конечным автоматом, управляемым данными (Data-Driven State Machine).

1.1. Состояние сущности
- Состояние хранится в битовой маске фиксированной (но расширяемой) длины.
- Маска разбита на семантические сегменты — непересекающиеся группы битов. Каждый сегмент соответствует конкретному аспекту сущности.
- Имена сегментов стандартизированы и образуются по схеме: `<IPO-функция>.<Домен>`, где IPO-функция — `input`, `process`, `output`, `state`, а Домен — `matter`, `energy`, `information`, `force` (и опциональные расширения: `magic`, `cyber` и т.п.).
- Значение каждого сегмента интерпретируется по универсальной шкале (например, 0–3 для 2-битных сегментов или 0–15 для 4-битных). Конкретная семантика значений задаётся словарём сегмента в JSON-конфигурации класса сущности.

1.2. Поведение сущности
- Поведение сущности определяется её текущим состоянием (значениями сегментов битовой маски) и входящим сигналом.
- Логика переходов между состояниями задаётся набором правил (правила реакции). Каждое правило имеет форму:
  ```
  WHEN <условие на сегменты маски> AND <тип сигнала>
  THEN <действие>
  ```
- Условия проверяют значения сегментов битовой маски (равенство, диапазон, сравнение с порогом). Действиями могут быть: изменение битовой маски (установка/сброс битов), генерация исходящего сигнала, изменение внутренних накопителей (напряжений).
- Правила загружаются из JSON-конфигурации класса сущности. Все экземпляры одного класса разделяют один и тот же набор правил (приспособленец).

1.3. Механика переключения состояний
- Для каждого сегмента битовой маски сущность может хранить накопитель напряжения $(U)$ (число с плавающей точкой).
- Непрерывные модели (калькуляторы) вычисляют дельты — изменения, которые прибавляются к соответствующим накопителям.
- При достижении накопителем абсолютного значения порога Шмитта $\theta$ биты сегмента переключаются дискретно. Пороги задаются в JSON-конфигурации класса или интерфейсов.
- Переключение бита сбрасывает соответствующий накопитель напряжения. Это обеспечивает гистерезис и устойчивость к флуктуациям.

1.4. Связи сущности
- Сущности связаны в иерархический двунаправленный граф.
- Каждая сущность может иметь:
  - ссылку на родительскую сущность (для обратной связи),
  - список дочерних сущностей (для прямой связи),
  - список одноранговых сущностей (для горизонтальных связей).
- Все взаимодействия опосредованы протоколами (см. раздел 2).

### 2. Протокол взаимодействия (Protocol)

Базовый принцип: протокол реализует паттерн Адаптер и обеспечивает преобразование сигналов между двумя сущностями с потенциально разными структурами битовых масок.

2.1. Назначение
- Принять сигнал от источника (в виде набора значений сегментов его маски).
- Преобразовать сигнал в формат, ожидаемый приёмником, в соответствии с правилами маппинга.
- Доставить преобразованный сигнал приёмнику через унифицированный интерфейс.

2.2. Свойства
- Stateless: протокол не хранит состояние отдельных экземпляров сущностей. Вся изменяемая информация (напряжения, пороги) хранится в самих сущностях.
- Единственный экземпляр на тип связи: для каждой пары классов сущностей (например, `Division → Brigade`) создаётся ровно один объект протокола. Этот объект обслуживает все пары экземпляров данных классов.
- Двунаправленность: один объект протокола содержит правила маппинга для обоих направлений: прямого (от родителя к потомку) и обратного (от потомка к родителю).

2.3. Маппинг сигналов
- Правила маппинга задаются в JSON-конфигурации протокола и определяют:
  - Имя сегмента источника (`source_segment`).
  - Имя сегмента приёмника (`target_segment`).
  - Тип трансформации: `direct` (прямое копирование), `linear` (линейное растяжение/сжатие шкалы), `threshold` (пороговое срабатывание), `aggregate` (агрегация от нескольких источников).
- Для агрегации (например, сбор откликов от многих дочерних сущностей) протокол может реализовывать накопление и пороговую логику в самой сущности-приёмнике, а протокол лишь доставляет индивидуальные сигналы.

2.4. Интерфейс
- Протокол взаимодействует с сущностями через унифицированный интерфейс:
  - `entity.get_segments(segment_names)` — возвращает значения указанных сегментов.
  - `entity.apply_signal(transformed_data)` — применяет преобразованные значения к сегментам сущности (запуская цепочку проверки правил и накопления напряжений).

### 3. Ключевые архитектурные принципы

3.1. Ориентированная на данные архитектура
Вся вариативная логика (структура состояния, правила поведения, протоколы связей) описана в JSON-конфигурациях. Ядро движка является универсальным интерпретатором этих конфигураций. Добавление новых классов сущностей, состояний или типов связей не требует изменения кода движка.

3.2. Разделение ответственности
- Сущность управляет своим состоянием, накоплением напряжений и переключением битов по порогам. Она является State Machine.
- Протокол управляет только преобразованием и передачей сигналов между сущностями. Он является Adapter.
- Калькуляторы (непрерывные модели) поставляют дельты для накопителей напряжений, работая на «аналоговом» уровне, в то время как сущности оперируют дискретными состояниями.

3.3. Производительность через приспособленца
Разделяемые объекты (правила поведения, протоколы) загружаются один раз при инициализации мира и существуют в единственном экземпляре на класс или на тип связи. Индивидуальные сущности хранят лишь битовую маску, накопители напряжений и ссылки на соседей, что обеспечивает минимальный расход памяти и высокую скорость обработки.

3.4. Расширяемость через конфигурацию
Моддер (или графический редактор контента) создаёт новые сущности и связи, описывая их в JSON. Для стандартных паттернов взаимодействия (например, «командир-подчинённый») используются шаблоны протоколов, которые автоматически применяются при совпадении имён сегментов. Для нестандартных связей достаточно явно прописать маппинг в конфигурации.
